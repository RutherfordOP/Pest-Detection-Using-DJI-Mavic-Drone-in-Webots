# Pest Detection Drone — Webots Simulation

A quadrotor drone simulation built in [Webots](https://cyberbotics.com/) that autonomously surveys farmland, detects pest-infected regions using camera vision, and logs geo-tagged results to JSON. Written entirely in Python.

> Built and tested on Windows with Webots R2023b and Python 3.11.

---

## What it does

The drone takes off, follows a lawn-mower grid pattern over a 200×200m farmland, and flags any brown/discoloured patches that indicate pest damage. If it encounters a tree or pole mid-flight, it sidesteps and continues. When the survey is done, it lands and dumps a `pest_detection_results.json` you can visualize with the included analysis script.

Sensors on board: GPS, Compass, LIDAR (obstacle detection), and a downward-facing Camera.

## Getting started

**1. Generate the world file**

```bash
cd C:\WebotsProjects\PestDetectionDrone
python create_world.py
```

This writes `worlds/farmland.wbt` — a 200×200m arena with a few trees and poles scattered around.

**2. Open the world in Webots**

`File → Open World → worlds/farmland.wbt`

You should see the arena, some brown cylinder obstacles, and the Mavic drone sitting at the bottom-left corner.

**3. Run the simulation**

Hit the **Play** button (green triangle). Watch the console for output like:

```
Phase 1: Generating coverage path...
Path generated with 120 waypoints
Phase 2: Starting navigation...
[Iteration 100] Position: [12.5, 12.3, 15.0], Pests detected: 0
```

**4. Check the results**

After the drone lands and the simulation ends, run:

```bash
python analyze_results.py
```

This plots pest locations over the farmland map and saves `pest_map.png`.

## How the path planning works

The coverage algorithm divides the farmland into a grid and flies the drone in a back-and-forth (lawn-mower) pattern at 15m altitude. It generates a list of (x, y, z) waypoints before takeoff — no real-time planning needed for the baseline path.

```
Grid: 200m × 200m, cell size 20m
Row 0: (10,10) → (190,10)
Row 1: (190,30) → (10,30)
...and so on
```

If LIDAR detects something within 5m, the drone shifts sideways 15m, waits a beat, then replans around the obstacle and picks up where it left off.

## Kinematics (simplified)

Webots handles the physics. The controller sends velocity commands rather than raw motor speeds. The `SimpleController` class does the inverse kinematics — given current position and a target waypoint, it computes the required velocity vector using proportional control (`kp = 0.15`). That velocity then maps to individual motor speeds across the four propellers.

Motor layout:
```
Front-Left (0)    Front-Right (1)
Rear-Left  (3)    Rear-Right  (2)
```

## Pest detection

Currently uses colour heuristics — the camera scans for brown/reddish pixels that suggest leaf damage. Real deployments would swap this out for a CNN or multispectral imaging pipeline. The `PestDetector` class in `pest_detector.py` is designed to make that swap easy.

## Common issues

| Problem | Fix |
|---|---|
| `Warning: Motor motor_1 not found` | Check motor names in `.wbt` match `motor_1` through `motor_4` |
| Position stuck at `[0, 0, 0]` | GPS not enabled — make sure simulation is actually running (press Play) |
| Drone doesn't lift off | Increase `base_thrust` from 50 to 70–80 in `drone_controller.py` |
| Simulation crashes | Reduce camera resolution or add a timeout to the main loop |

## Extending this

A few directions worth exploring:

- **ML pest detection** — train a small CNN on real pest-damage leaf images, drop it into `pest_detector.py`
- **Multi-drone coverage** — `MultiDroneController` class is already stubbed out, just needs multiple Robot nodes in the `.wbt`
- **RRT\* / A\* planning** — replace the fixed grid with proper motion planning for complex environments
- **ROS bridge** — Webots has a ROS2 interface if you want to connect this to real hardware
